## 1. 제품 개요

### 1.1 한 줄 정의
중소·창업기업이 자사에 맞는 **정부지원사업을 자동으로 발굴**하고, **합격 가능성 높은 사업을 추천**받아, **공고 양식에 맞는 제안서 초안을 AI로 작성**하기까지를 한 흐름으로 처리하는 SaaS.

### 1.2 핵심 파이프라인
```
[1단계 발굴]        [2단계 선정]         [3단계 제안서]
크롤링 기반    →    기업↔공고 매칭    →   양식 자동 채움
공고 수집·정규화     적합도 스코어링       + AI 초안 생성
```

### 1.3 차별점 (가설)
- 단순 공고 검색(기업마당 등)을 넘어, **우리 회사 프로필 기반 맞춤 추천 + 제안서 작성**까지 연결.
- 기업이 매번 공고를 뒤지고, 양식을 새로 채우는 반복 노동을 제거.

## 2. 대상 사용자 / 페르소나

주 사용자: **다수 기업이 가입해 각자 사용하는 SaaS** 형태.

| 페르소나 | 설명 | 핵심 니즈 |
|---------|------|----------|
| 스타트업 대표/기획자 | 인력 부족, 지원사업 정보에 어두움 | "우리가 받을 만한 사업을 알아서 찾아줘" |
| 중소기업 경영지원 담당 | 여러 공고를 동시 관리 | 마감·요건 관리, 제안서 작성 부담 경감 |
| (확장) 컨설턴트 | 여러 고객사 대행 | *현재 범위 밖, 향후 확장 고려* |

### 2.1 1차 타깃 (확정)
**IT/SW 스타트업 우선.** 전 업종을 처음부터 받지 않고, IT/소프트웨어 분야 창업·초기기업에 집중하여 매칭 정확도를 빠르게 끌어올린다.

- **업종**: 정보통신·SW·플랫폼·AI·SaaS 등 IT 계열 우선
- **업력**: 창업 ~ 7년 이내(초기 스타트업) 중심
- **규모**: 중소·소상공인 범위
- 전 업종 확장은 매칭 엔진이 안정화된 이후의 로드맵 항목으로 둔다.

> 의미: 2단계 매칭의 기준 데이터(공고 `target`, 기업 프로필)와 추천 가중치를 **IT/SW 맥락에 최적화**한다 (예: R&D·기술개발·SW 바우처·청년창업 계열 공고 가중).

## 3. 문제 정의 & 가치 제안

### 3.1 문제
1. 지원사업 공고가 **수십 개 기관에 분산** (기업마당, K-Startup, 지자체, TIPA, 창업진흥원 등).
2. 공고마다 **자격요건·양식·평가기준이 제각각** → 적합 여부 판단에 시간 소요.
3. 제안서/사업계획서를 **매번 처음부터** 작성, 양식 맞추기에 노동 집중.

### 3.2 가치 제안
- **발굴 자동화**: 흩어진 공고를 한곳에 수집·정규화.
- **선정 지원**: 우리 회사 프로필로 적합도 점수화 → 지원할 사업 선별.
- **작성 가속**: 공고 양식 자동 인식 + 회사 정보 자동 채움 + AI 초안.

## 5. 기능 요구사항 — 【1단계: 발굴(크롤링)】

### 5.1 목표
분산된 정부지원사업 공고를 주기적으로 수집하여, 통일된 스키마로 저장하고 검색 가능하게 한다.

### 5.2 수집 대상 (1차 후보)
| 소스 | 성격 | 공식 API 유무 |
|------|------|-------------|
| 기업마당 bizinfo.go.kr | 중소기업 지원 통합 | 있음(공공데이터포털) — fallback 권장 |
| K-Startup | 창업 지원 | 있음 — fallback 권장 |
| 공공데이터포털 data.go.kr | 부처 공고 | 있음 |
| 지자체/유관기관(TIPA 등) | 분산 공고 | 일부 없음 → 크롤링 |

> **메인 전략(사용자 결정): 크롤링 위주.**
> 단, 공식 API가 있는 소스는 안정성을 위해 **API 우선 적용을 권고**(리스크 8.x 참조).

### 5.3 수집 데이터 표준 스키마 (정규화)
모든 소스를 아래 공통 필드로 정규화한다.

| 필드 | 설명 | 예시 |
|------|------|------|
| `id` | 내부 고유 ID | gov-2026-00123 |
| `source` | 출처 기관 | bizinfo |
| `title` | 공고명 | 2026 창업도약패키지 |
| `category` | 분야(자금/R&D/판로/인력 등) | R&D |
| `target` | 지원 대상(업종·업력·규모) | 3년 이내 창업기업 |
| `region` | 지역 제한 | 전국 / 서울 |
| `amount` | 지원 금액 | 최대 1억 |
| `period_start` / `period_end` | 접수 시작/마감일 | 2026-07-01 / 2026-07-31 |
| `attachments` | 첨부(공고문·양식 HWP/PDF) | URL 리스트 |
| `source_url` | 원문 링크 | https://... |
| `crawled_at` | 수집 시각 | timestamp |

### 5.4 기능 요구사항 (FR)
- **FR-1.1** 소스별 크롤러를 플러그인 구조로 추가/관리할 수 있다.
- **FR-1.2** 수집 주기를 스케줄링한다(예: 1일 1~수회).
- **FR-1.3** 중복 공고를 감지·병합한다(제목·기관·마감일 기준 디둡).
- **FR-1.4** 첨부된 공고문/양식 파일(HWP·PDF·DOCX)을 다운로드·보관한다(3단계에서 양식 파싱에 사용).
- **FR-1.5** 마감 임박/신규 공고를 플래그로 표시한다.
- **FR-1.6** 수집 실패(사이트 개편 등)를 로깅·알림한다.
- **FR-1.7** 공고 검색·필터(분야·지역·대상·마감일) 제공.

### 5.5 비기능 요구사항 (NFR)
- **NFR-1.1** 크롤링 시 robots.txt·이용약관 준수, 요청 rate-limit 적용.
- **NFR-1.2** 사이트 구조 변경에 대비해 파서를 소스별로 격리(한 곳 깨져도 전체 중단 없음).
- **NFR-1.3** 원문 보관으로 추적성(공고 변경 이력) 확보.

## 6. 기능 요구사항 — 【2단계: 선정 — 매칭·적합도 스코어링】

### 6.1 목표
1단계에서 수집·정규화한 공고와 **가입 기업의 프로필**을 대조하여, 각 공고의 **적합도(0~100점)** 를 산출하고 "지원할 만한 사업"을 추천한다. 1차 타깃이 IT/SW 스타트업이므로 해당 맥락에 가중치를 최적화한다.

### 6.2 기업 프로필 스키마 (매칭 입력)
가입 시/온보딩에서 1회 입력, 이후 재사용. **이 데이터가 3단계 제안서 자동 채움에도 그대로 쓰인다.**

| 필드 | 설명 | 예시 |
|------|------|------|
| `company_name` | 회사명 | (주)리아솔 |
| `biz_no` | 사업자번호 | 123-45-67890 |
| `industry` | 업종(표준산업분류 기반) | 소프트웨어 개발 |
| `tech_keywords` | 기술 키워드(태그) | AI, SaaS, 데이터분석 |
| `founded_date` | 설립일 → 업력 자동계산 | 2023-03 |
| `employee_count` | 상시 근로자 수 | 8 |
| `revenue` | 매출 규모 | 3억 |
| `region` | 소재지 | 서울 |
| `certifications` | 보유 인증(벤처·이노비즈·연구소 등) | 벤처기업, 기업부설연구소 |
| `prior_support` | 기수혜 이력(중복지원 배제 판단) | 2025 청년창업사관학교 |
| `target_needs` | 희망 지원 유형 | R&D 자금, 판로개척 |

### 6.3 매칭 로직 (적합도 스코어링)
공고 `target`/요건과 기업 프로필을 **규칙 기반(필터) + 가중 점수(스코어)** 2단계로 평가.

**(A) 자격 필터 — 통과/탈락 (Hard filter)**
하나라도 불충족 시 추천 제외(또는 "자격 미달"로 별도 표시).
- 업력 조건 (예: 창업 3년 이내) ↔ `founded_date`
- 지역 제한 ↔ `region`
- 기업 규모/업종 제한 ↔ `industry`, `employee_count`
- 접수 마감 미경과 ↔ `period_end`
- 중복지원 제한 ↔ `prior_support`

**(B) 적합도 점수 — 가중 합산 (Soft score, 0~100)**
| 평가 축 | 가중치(초기 가설) | 산출 방식 |
|---------|-----------------|----------|
| 기술/분야 적합 | 35% | 공고 분야·키워드 ↔ `tech_keywords` 유사도 (IT/SW 가중) |
| 자격 적합도 | 25% | 요건 충족 정도(여유 클수록 가점) |
| 지원 규모 매력 | 15% | `amount` 대비 기업 규모 적정성 |
| 마감 여유 | 10% | 준비 가능 기간(임박할수록 감점) |
| 경쟁/선정가능성 | 15% | (확보 가능 시) 경쟁률·기수혜 적합 패턴 |

> 가중치는 운영 데이터로 보정. 초기엔 키워드 매칭 + 규칙 기반으로 시작하고, 데이터가 쌓이면 학습형으로 고도화.

### 6.4 기능 요구사항 (FR)
- **FR-2.1** 기업 프로필을 온보딩에서 입력·수정할 수 있다.
- **FR-2.2** 신규/갱신 공고에 대해 가입 기업별 적합도를 자동 산출한다.
- **FR-2.3** 적합도 순 정렬 + 자격 미달/마감임박 배지로 추천 리스트를 제공한다.
- **FR-2.4** 점수의 **근거를 설명**한다(왜 추천했는지 — 충족 요건/매칭 키워드 표시).
- **FR-2.5** 사용자가 "관심/지원예정/제외"로 공고를 분류·관리(파이프라인 보드)한다.
- **FR-2.6** 마감 임박·신규 적합 공고를 알림(이메일/대시보드)한다.

### 6.5 비기능 요구사항 (NFR)
- **NFR-2.1** 추천 근거의 투명성 — 블랙박스 점수 금지, 충족/미충족 항목을 명시.
- **NFR-2.2** 멀티테넌트 — 기업별 프로필·추천 결과 격리.
- **NFR-2.3** 공고 수가 늘어도 기업별 스코어링이 일괄 배치로 처리 가능.

### 6.6 조건 입력 기반 맞춤 검색 (On-demand 검색)
저장된 프로필로 자동 추천(6.2~6.5)되는 것과 별개로, 사용자가 **조건을 직접 입력해 즉석에서** 적합 사업을 검색하는 기능. (프로필 미입력 상태, 혹은 "이번 건은 다른 조건으로 찾아보고 싶다"는 상황 대응)

**(A) 입력 항목**
| 입력 | 유형 | 설명 |
|------|------|------|
| 지역 | 선택 | 전국/시·도 (공고 `region` 매칭) |
| 년차(업력) | 숫자/구간 | 설립 후 경과 연수 (창업 N년 이내 요건 매칭) |
| 분야 | 선택(다중) | 업종·지원분야 (R&D/자금/판로/인력 등, IT/SW 우선 가중) |
| **회사 특징** | **자유 서술(자연어)** | "AI 기반 물류 SaaS, 특허 2건, 수출 준비 중" 등 자유 입력 |

**(B) 동작 — 정형 필터 + 의미 검색 결합**
1. **정형 조건(지역·년차·분야)** → 자격 필터(6.3-A)로 부적합 공고 제외.
2. **회사 특징(자유 서술)** → 임베딩/의미 유사도 기반으로 공고 내용과 매칭(키워드 단순일치 한계 보완).
3. 두 결과를 결합해 **적합도 순으로 랭킹** + 추천 근거 표시(6.3-B, FR-2.4 재사용).

> 회사 특징 자유 서술은 AI가 **핵심 키워드·기술·니즈를 추출**해 검색에 활용. 이렇게 추출된 정보는 사용자가 동의하면 **기업 프로필(6.2)로 저장**해 재사용 가능 → 2단계 자동 매칭/3단계 작성으로 자연스럽게 연결.

**(C) 기능 요구사항 (FR)**
- **FR-2.7** 지역·년차·분야 조건과 회사 특징 자유 서술을 입력받는 검색 폼을 제공한다.
- **FR-2.8** 정형 조건은 필터로, 자유 서술은 의미 검색으로 결합해 적합 사업을 랭킹한다.
- **FR-2.9** 조건 일부만 입력해도(예: 지역만) 동작하며, 조건이 좁아 결과가 적으면 **조건 완화 제안**을 한다.
- **FR-2.10** 검색 조건을 저장/재실행하고, 추출된 특징을 기업 프로필로 승격 저장할 수 있다.

## 7. 기능 요구사항 — 【3단계: 제안서 — 양식 자동 채움 + AI 초안】

### 7.1 목표
선정한 공고에 대해, 공고 양식을 **웹 서식으로 재구성**하여 작성하고, **회사 프로필 정보를 자동으로 채운 뒤**, **AI가 공고 요건 + 평가기준에 맞춘 초안**을 생성한다. 사용자는 검토·수정 후 제출용 파일(DOCX/PDF)로 내려받는다.

### 7.2 양식 처리 방식 (확정)
**웹 서식 작성 후 다운로드.**
- 공고 첨부 양식(HWP/PDF)에서 **항목 구조를 추출**해 웹 폼으로 재구성.
- 작성은 브라우저 웹 폼에서 진행, 제출용은 **DOCX/PDF로 내보내기**.
- **HWP 직접 입출력은 로드맵 후순위** (국내 현실상 필요하나 난이도·8.3 리스크로 2차).

> 초기엔 공고별 양식을 수기/반자동으로 "양식 템플릿"화해 등록하는 방식 허용 (완전 자동 파싱은 점진 고도화).

### 7.3 AI 초안 생성 — 그라운딩 (확정)
초안의 근거 데이터로 **3가지를 모두 활용**한다:
1. **회사 프로필** (6.2) — 회사 소개·역량·실적·인증
2. **공고 요건** (5.3) — 지원 목적·대상·지원 내용
3. **평가기준** — 공고문에서 추출한 심사 항목·배점

→ AI는 단순 작문이 아니라 **"이 평가기준에서 점수를 받도록"** 항목별 내용을 구성한다. (예: 평가에 "시장성 30점"이 있으면 시장성 단락을 강화)

| 입력 | 출처 | 역할 |
|------|------|------|
| 회사 프로필 | 2단계 입력값 | 사실 기반 자동 채움 (지어내지 않음) |
| 공고 요건 | 1단계 수집 | 작성 방향·요구 항목 |
| 평가기준·배점 | 공고문 파싱 | 단락별 강조점·분량 배분 |

### 7.4 작성 흐름 (UX)
```
공고 선택 → 양식 템플릿 로드 → 회사정보 자동 채움
   → AI 초안 생성(항목별) → 사용자 검토·수정 → DOCX/PDF 내보내기
```

### 7.5 기능 요구사항 (FR)
- **FR-3.1** 공고 양식을 항목 단위 웹 서식으로 제공한다.
- **FR-3.2** 회사 프로필 기반 공통 항목(회사 개요·대표자·인증 등)을 자동 채운다.
- **FR-3.3** 항목별로 AI 초안을 생성하고, 사용자가 부분 재생성할 수 있다.
- **FR-3.4** AI는 **평가기준의 배점에 맞춰** 항목별 분량·강조를 조정한다.
- **FR-3.5** 사실 정보(수치·실적)는 **프로필 입력값만 사용**하고, 미입력 항목은 비워두거나 사용자에게 입력을 요청한다(환각 방지).
- **FR-3.6** 작성본을 DOCX/PDF로 내보낸다.
- **FR-3.7** 작성 중 자동 저장·버전 관리(제출 전 되돌리기).

### 7.6 비기능 요구사항 (NFR)
- **NFR-3.1** LLM은 **Claude API**(예: `claude-opus-4-8` 또는 `claude-sonnet-4-6`) 사용 가정. 모델·비용은 별도 확정.
- **NFR-3.2** 회사 민감정보(사업자번호·매출 등)는 프롬프트 전송 범위를 최소화·통제.
- **NFR-3.3** AI 산출물은 항상 **사람 검토 후 제출**을 전제(자동 제출 금지).

## 9. SaaS 운영 구조

다수 기업 대상 멀티테넌트 SaaS 전제.
- **테넌시**: 기업(테넌트)별 프로필·공고 추천·작성 문서 격리.
- **회원/권한**: 기업 단위 가입, 구성원 초대(관리자/작성자/뷰어 역할).
- **과금(가설)**: 프리(공고 열람) → 유료(맞춤 추천·AI 제안서 작성 횟수/시트 기반). 구체안은 별도.
- **데이터 보안**: 사업자정보·제안서는 테넌트 격리 저장, AI 전송 데이터 최소화(7.6 NFR).
